## 한 줄 정의

모두가 가능한 시간을 찾지 않고, **아무도 부담스럽지 않은 시간을 남기는** 회의 일정 도구.

## 기존 서비스와의 차별점

| | 기존 도구 (when2meet · Doodle · 되는시간 등) | 사이 (Sai) |
|---|---|---|
| **질문** | "언제 가능하세요?" | "언제가 부담스러우세요?" |
| **데이터 모델** | 가용성 수집 → 교집합 탐색 | 거부(veto) 수집 → 부담의 합집합 제거 |
| **입력 비용** | 가능한 칸을 전부 칠함 (수십 셀) | 싫은 칸만 지움 (몇 번의 탭) |
| **침묵의 해석** | 무응답·전부 가능 = 동의 | 거부만 신호 — 침묵을 동의로 처리하지 않음 |
| **UI 상태** | 가능(초록)/불가(빨강) | **'가능' 상태가 없음** — 중립/거부 2states |
| **익명성** | 응답자 이름 공개가 기본 | 사유·숫자만 집계, 주최자도 못 봄. 채팅도 동물 이름 |
| **결정 방식** | 히트맵을 보고 주최자가 눈으로 탐색 | 필수 전원 가능 + 최소 거부 시간 1~3개 자동 추천 |
| **참석자 모델** | 전원 동급 | 필수/선택 구분 — 선택자로 인한 무한 지연 방지 |
| **숨은 제약** | 캘린더 빈칸 = 가능으로 간주 | "점심 직후·외근 이동" 등 캘린더 밖 제약을 사유 태그로 포착 |
| **재식별 방어** | 응답자·응답 시점 공개 → "누가 방금 냈나 × 집계 변화" 대조로 특정 가능 | 개인별 응답 여부 비공개(집계만) + 응답 배치 반영 + 알림 시점 정보 제거 — 타이밍 상관관계 차단 |

## 근거 데이터 (자체 설문 · N=26)

- **84.6%** — 곤란한데 눈치 때문에 "괜찮다"고 한 적 있다
- **50%** — 사실 내가 불편했지만 그냥 참았다 → *가짜 합의의 직접 증거*
- **50%** — "안 되는 시간 표시가 더 편하다" (되는 시간 15.4%의 3배)
- **88.4%** — 회의를 단톡방·윗사람으로 정함 (도구 사용 11.5%) → *기존 도구가 못 푼 시장 공백*

## 지원 어필 (1,979자)

사이(Sai)는 회의 시간을 '찾는' 앱이 아니라, 회의를 둘러싼 침묵을 '푸는' 앱입니다.

시중의 모든 일정 조율 서비스 — when2meet, Doodle, 되는시간, 캘린더 연동 스케줄러 — 는 사실 하나의 같은 질문을 합니다. "언제 가능하세요?" 저는 이 질문 자체가 틀렸다고 생각했습니다. 자체 설문(N=26)에서 84.6%가 "곤란한 시간인데 눈치 때문에 괜찮다고 한 적이 있다"고 답했고, 50%는 "사실 내가 불편했지만 그냥 참았다"고 했습니다. 모두가 '가능'이라고 답한 시간조차 누군가는 조용히 참고 있는 가짜 합의였던 겁니다. 기존 도구는 조율의 표면을 자동화했을 뿐, 진짜 마찰 — 말 못 하는 부담 — 은 건드리지 못했습니다. 88.4%가 여전히 단톡방과 윗사람의 지시로 시간을 정하고, 도구 사용률이 11.5%에 그치는 이유가 여기에 있습니다.

그래서 사이는 질문을 뒤집었습니다. "언제 가능하세요?"가 아니라 "언제가 부담스러우세요?"를, 그것도 익명으로 묻습니다. 이 반전이 만드는 차별점은 네 가지입니다.

첫째, 데이터 모델의 반전입니다. 기존 도구는 가능한 시간의 '교집합'을 찾지만, 사이는 부담의 '합집합'을 제거합니다. 캘린더의 빈칸이 곧 가능한 시간은 아닙니다. 점심 직후의 나른함, 외근 뒤의 이동, 집중이 필요한 오전 — 캘린더에 적히지 않는 제약은 가용성 데이터에 잡히지 않습니다. 거부는 이 숨은 제약까지 담아냅니다. 입력 비용도 다릅니다. 수십 개의 빈칸을 칠하는 대신 싫은 칸 몇 개만 지우면 끝나고("안 되는 시간 표시가 편하다" 50%로, 되는 시간 표시 15.4%의 3배), 입력이 가벼우니 응답률이 오릅니다.

둘째, 침묵을 동의로 처리하지 않습니다. 기존 도구에서 '전부 가능'은 협조로 보이지만, 위계 앞에서는 체념일 때가 많습니다. 사이는 거부만을 신호로 취급하기에, 괜찮은 척을 데이터로 강요하지 않습니다. 그래서 UI에도 '가능(초록)' 상태가 아예 없습니다. 중립과 거부, 두 상태뿐입니다.

셋째, 익명을 기능이 아니라 구조로 설계했습니다. 거부는 "3명이 부담스러워해요 (점심 직후 2, 외근 1)"처럼 사유와 숫자로만 집계되고, 누가 지웠는지는 주최자조차 볼 수 없습니다. 회의별 채팅도 동물 이름으로만 참여하며, 확정 화면에서는 "누가 어떤 시간을 지웠는지는 끝까지 비밀이에요"라고 한 번 더 약속해 익명이라는 신뢰를 플로우의 끝까지 유지합니다. 익명은 만능이 아니라 일부에게 반드시 필요한 안전장치이고, 이 심리적 안전이 있어야 진짜 제약이 드러나 진짜 합의가 가능해집니다.

넷째, 결정을 압축합니다. 사이는 필수 참석자 전원이 가능하면서 거부가 가장 적은 시간을 1~3개로 추천하고, 주최자가 확정합니다. 필수·선택 구분 덕에 선택 참석자 한 명 때문에 회의가 무한정 밀리지 않고, 주최자의 결정 비용은 표가 아니라 추천 카드 하나로 줄어듭니다. 추천이 아쉬우면 전체 시간대의 거부 현황을 농도 그래프로 투명하게 열어볼 수도 있습니다.

이 개념은 기획서로 멈추지 않았습니다. 회의 생성부터 초대, 지우기, 익명 집계, 추천, 확정, 캘린더 추가까지 전 플로우가 실제로 동작하는 프로토타입으로 구현해 검증했고, 화면 설계 스펙과 브랜드 시스템 — 프로스티드 글라스 'S2' 로고, 커스텀 라인 아이콘 24종 — 까지 하나의 언어로 정리했습니다. 문구에도 같은 원칙을 담았습니다. '거부'라는 예민한 행동을 다루는 앱이기에, 부정 대신 긍정으로 — "사유 없이 지워도 괜찮아요", "모든 시간 괜찮아요" — 마지막 순간까지 부담을 덜어내는 라이팅을 썼습니다.

토스가 공인인증서라는 '당연한 마찰'을 지워 송금을 다시 정의했듯, 사이는 일정 조율의 당연한 마찰 — 눈치 — 를 지웁니다. 화면을 더하는 대신 입력을 뺐고, 빈 시간 칠하기·양보 횟수 추적·일정을 정하는 채팅을 의도적으로 제외했습니다. 문제를 사용자의 편에서 다시 정의하고, 빼기로 완성하는 것. 사이가 지우는 것은 결국 시간이 아니라 말하지 못하던 부담이고, 그것이 토스의 방식으로 만든 저의 답이라고 믿습니다.
